The octo-auth wire contract defines **3 verify endpoints** that resolve a credential to a Principal. Both the [Go and TypeScript SDKs](/reference/sdks) implement these, and `octo-server` serves them. This page is generated from the contract, so it never drifts.

<Tip>
  You normally don't call these directly — use the [octo-auth SDK](/guides/integrators/verify-credentials-with-octo-auth), which wraps them behind a `MultiVerifier`, adds caching, and fails closed. See the [Security & auth model](/concepts/security-and-auth) for the credential realms.
</Tip>

## sessionVerifier — verify a human session token

`POST /v1/auth/verify` · `verifySession`

Resolves a human web-session token (32-hex UUID, no prefix) to a Principal. With `?include=context`, also returns the user's `spaces` and `owned_bots_by_space` so downstream middleware can fail-closed on `X-Space-Id` mismatches without a second round-trip. See the top-level "include=context semantics" section for the state table and the fail-closed contract.

**Query parameters**

| Name | In | Description |
|---|---|---|
| `include` | query | When set to `context`, the response is populated with `context_included`, `spaces`, and `owned_bots_by_space`. Any other value is ignored by the server. |

**Request body**

| Field | Type | Required | Description |
|---|---|---|---|
| `token` | string | ✓ | Human session token — 32 hex characters, no prefix. The SDK classifies "no known prefix" as sessionVerifier realm. |

**Response (200)**

| Field | Type | Required | Description |
|---|---|---|---|
| `uid` | string | ✓ | Principal UID. |
| `name` | string | ✓ |  |
| `role` | string | ✓ | System role. Empty string for regular users; other values (e.g. `"admin"`, `"superAdmin"`) are platform-defined. |
| `owned_bots` | —[] |  | Bots owned by this user. Present in v2+ responses (may be an empty array); may be omitted by pre-v2 servers. Each entry is an `OwnedBot` object with `uid` and `name`. |
| `context_included` | boolean |  | Present-and-`true` iff `?include=context` was sent AND the v2+ server populated the extra fields. Absent otherwise — see the top-level "include=context semantics" section for the wire state table. |
| `spaces` | string[] |  | Space IDs the user is a member of. Authoritative when and only when `context_included == true`. An empty array is a legitimate v2+ response for a user with zero memberships. |
| `owned_bots_by_space` | object |  | Map of space_id → list of bot UIDs owned by this user in that space. Authoritative when `context_included == true`. |

## botTokenVerifier — verify a bot token

`POST /v1/auth/verify-bot` · `verifyBotToken`

Resolves a bot token to a Principal. Prefix routes: - `bf_&lt;...&gt;` — User Bot. - `app_&lt;...&gt;` — App Bot. **Reserved**; SDK v1 does not implement   the App Bot wire path. Servers that receive an `app_` token   without App Bot support MUST respond `AUTH_INVALID_CREDENTIAL`. `space_id` in the response is server-authoritative — see the top-level "Anti-spoof space_id" section. Consumer middleware MUST NOT read `X-Space-Id` from the request for bot-authenticated traffic. This endpoint does **not** accept `?include=context`.

**Request body**

| Field | Type | Required | Description |
|---|---|---|---|
| `bot_token` | string | ✓ | Bot token. Accepted prefixes: - `bf_&lt;...&gt;` — User Bot; SDK v1 wire path. - `app_&lt;...&gt;` — App Bot; **reserved for a future release**,   not implemented by SDK v1. Servers without App Bot support   MUST respond `AUTH_INVALID_CREDENTIAL` for `app_` tokens. Unprefixed tokens are rejected by the SDK's `MultiVerifier` classifier before the request is sent. |

**Response (200)**

| Field | Type | Required | Description |
|---|---|---|---|
| `bot_uid` | string | ✓ |  |
| `bot_name` | string | ✓ |  |
| `role` | string |  | Optional bot role hint. Server does not emit this field as of v3, so consumers always observe empty string; declared here so SDKs can decode a future server that supplies it without a wire-shape bump. |
| `owner_uid` | string | ✓ |  |
| `owner_name` | string | ✓ |  |
| `space_id` | string | ✓ | Server-authoritative space binding for this bot. Consumer middleware MUST ignore any `X-Space-Id` header from the request and treat this value as the canonical binding — see the top-level "Anti-spoof space_id" section. |

## userKeyVerifier — verify a user API key

`POST /v1/auth/verify-api-key` · `verifyUserKey`

Resolves a `uk_&lt;...&gt;`-prefixed user API key (length &gt;= 35) to a Principal. Long-lived keys are used by daemons and CI runners (e.g. octo-fleet's runtime daemon). Accepts the same `?include=context` query parameter as `sessionVerifier`; the same state table applies. Unlike session, an api key is bound to exactly one space, so `owned_bots` on the response is a single-key map (`&#123; &lt;bound_space_id&gt;: [...bot_uids] &#125;`).

**Query parameters**

| Name | In | Description |
|---|---|---|
| `include` | query | When set to `context`, the response is populated with `context_included` and `owned_bots` (single-key map on the api key's bound space). |

**Request body**

| Field | Type | Required | Description |
|---|---|---|---|
| `api_key` | string | ✓ | User API key with `uk_` prefix; total length MUST be &gt;= 35. The SDK's `MultiVerifier` enforces the length floor before dispatching to this endpoint. |

**Response (200)**

| Field | Type | Required | Description |
|---|---|---|---|
| `uid` | string | ✓ | Principal UID that owns the key. |
| `space_id` | string | ✓ | The single space this api key is bound to. **Server-authoritative** — consumer middleware MUST fail-closed when this value is empty. Not shaped as a list because an api key binds to exactly one space (unlike a session, which spans all spaces the user belongs to). |
| `context_included` | boolean |  | Present-and-`true` iff `?include=context` was sent AND the v2+ server populated `owned_bots`. Absent otherwise — see the top-level "include=context semantics" section. |
| `owned_bots` | object |  | Map of `space_id` → bot UIDs the key's owner controls in that space. Populated only when `?include=context` was requested and the server honoured it. Because api keys bind to exactly one space, the outer map has at most one key — the `space_id` above. Consumers derive the `Spaces` slice on the SDK Principal from the sorted keys of this map. |
