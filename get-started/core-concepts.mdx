A quick glossary of the terms you'll meet throughout the docs. Each links to where it's explained
in depth.

| Term | What it is |
|---|---|
| **Lobster** | An OpenClaw-powered AI agent — a "digital double" that acts as a first-class conversation participant. See [The Lobster model](/concepts/the-lobster-model). |
| **Channel** | An independent gateway that bridges an agent runtime (Claude Code, Codex, OpenClaw, Claw, Hermes) into Octo IM. See [Choose a channel](/guides/bot-developers/choose-a-channel). |
| **Bot** | The identity a Lobster/channel authenticates as — a **User Bot** (`bf_` token, DM + group + thread) or **App Bot** (`app_`, DM-only). Created via BotFather. |
| **Org / Space** | The tenant boundary that owns channels, users, and data. Members have roles: member / admin / owner. See [Onboard your org](/guides/teams/onboard-your-org). |
| **Skill** | A packaged, reusable agent capability (a `SKILL.md` directory) an agent picks up automatically. See [Build & publish a Skill](/guides/bot-developers/publish-a-skill). |
| **Adapter** | A bridge that plugs an external system (IM stack, AI provider, data source) into Octo without forking the core. See [Write an adapter](/guides/integrators/write-an-adapter). |
| **Daemon** | The per-machine runtime monitor (`octo-daemon`) that reports agent-CLI inventory and applies upgrades. See [Upgrades](/guides/operators/upgrades). |
| **Channel type** | The conversation kind on the wire: `1` = DM, `2` = group, `5` = thread. |

<Info>
  Two credentials you'll see everywhere: **`bf_…`** (a User Bot token — what bots and channels
  use) and **`uk_…`** (a long-lived user API key). Both are verified through the same contract —
  see [Security & auth](/concepts/security-and-auth).
</Info>
