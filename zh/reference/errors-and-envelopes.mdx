## Wire vs. SDK error kinds

The SDK exposes five typed error kinds
(`ErrKindInvalidCredential` / `ErrKindDisabled` /
`ErrKindInfraFailure` / `ErrKindForbidden` / `ErrKindPreV2Server`).
Four of them map onto wire codes; `ErrKindPreV2Server` is internal
and never appears on the wire — the SDK derives it from an absent
`context_included` field on an otherwise-successful response and
surfaces it via `ContextUnknownServer` on the returned Principal.

| SDK error kind              | Wire code                  | HTTP | Notes                                             |
| --------------------------- | -------------------------- | ---- | ------------------------------------------------- |
| ErrKindInvalidCredential    | `AUTH_INVALID_CREDENTIAL`  | 401  | Anti-enumeration catch-all — see below.           |
| ErrKindDisabled             | `AUTH_DISABLED`            | 403  | Principal authenticated but disabled.             |
| ErrKindForbidden            | `AUTH_FORBIDDEN`            | 403  | Authenticated but not authorised for this space.  |
| ErrKindInfraFailure         | `AUTH_INFRA_FAILURE`        | 503  | Upstream verify endpoint unavailable.             |
| ErrKindPreV2Server          | *(not on the wire)*        | —    | Internal signal; see design docs.                 |

## Anti-enumeration

`AUTH_INVALID_CREDENTIAL` collapses every "token bad" reason —
missing, malformed, unknown, expired, disabled at the credential
layer, kind mismatch — into a single wire outcome. The specific
reason lives in the server's audit log only. Callers MUST NOT
branch on any implied sub-reason; the SDK's typed error preserves
only the bucket (`ErrInvalidCredential`), never the finer reason.
