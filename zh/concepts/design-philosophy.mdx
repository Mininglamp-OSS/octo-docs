Octo 遵循三条共同原则发布，它们适用于生态中的**每一个仓库**。这不是营销话术——而是整体架构呈现如今样貌的根本原因。

## 1. 本地优先

凡是能在用户自己机器上运行的——聊天、嵌入、智能体——都应当如此。你的数据始终归你所有；云是一种选择，而非必需。

<Info>
  这正是[官方部署](/zh/guides/operators/deploy-compose)采用自包含单主机 Compose 栈（server + IM + MySQL + Redis + MinIO）的原因，也是机器人频道作为独立本地网关运行的原因，还是 `octo-cli` 是一个不依赖任何自有服务的单一静态二进制的原因。
</Info>

## 2. 人类判断，AI 思考并执行

人类专注于**品味**——什么重要、什么正确、该发布什么。**智能体**承担*思考*与*执行*的负担。智能体是一等对话参与者，而非附加上去的聊天机器人。

关于这在服务端智能体编排中的实现方式，参见 [智能体模型](/zh/concepts/the-agent-model)。

## 3. 发布即产品

每一次开源切分都以自包含的**产品**形态发布，而非代码堆砌：每次发布一次 squash，采用 Apache-2.0 许可，无内部包袱，仅凭仓库本身即可复现。

<Note>
  这也是这套文档所遵循的契约：[参考文档](/zh/reference/rest-websocket-api)由随每次发布*内附*的规范生成，因此你读到的内容始终与你能运行的相符。
</Note>
