Octo 通过两个部件让一整队本地运行时（Claude Code、OpenClaw、Codex、Hermes 守护进程）
保持最新：**`octo-version-sync`** 发布唯一的版本事实来源，而
**`octo-daemon-cli`** 运行在每台机器上，负责上报清单并应用升级。

## octo-version-sync —— 唯一的 version.json

**[`octo-version-sync`](https://github.com/Mininglamp-OSS/octo-version-sync)** 是一个长期运行的
服务，它按可配置的间隔轮询 **GitHub Releases**（Go 二进制文件）和 **npm**（OpenClaw 插件），
对每个 release 进行归一化，并将单一的 `version.json` 原子地写入
对象存储：

```json
{ "latest_version": "…", "release_meta": { "assets": [] }, "checksums": {} }
```

它追踪 `octo-daemon`、`claude`、`codex`、`hermes`、`openclaw` 以及 `octo` 插件——编辑
`components.json` 即可更改集合（无需重新部署）。`POST /trigger`（Bearer token）可强制发起一次扫描。

<Info>
  **柔性容错：** 若某个组件的轮询失败，它会保留先前的数据并将该条目标记为
  `status: stale`；整体失败则会完全跳过写入——守护进程绝不会看到写了一半的
  事实来源。
</Info>

## octo-daemon-cli —— 每台机器的监视器

**[`octo-daemon-cli`](https://github.com/Mininglamp-OSS/octo-daemon-cli)** 运行在每台承载
agent CLI 的机器上：

```bash
npm install -g @mininglamp-oss/octo-daemon
octo-daemon config --server-url <fleet-url> --api-key uk_xxx
octo-daemon run        # foreground
octo-daemon start      # background via pm2
```

`config` 会针对 `POST <fleet-url>/v1/runtimes/verify` 校验密钥并解析
`space_id`。守护进程探测 Claude Code 和 OpenClaw CLI，将清单推送给
[`octo-fleet`](/zh/ecosystem/repository-guide)，并支持从 Octo web UI 发起的**一键远程升级**
（原子化服务端认领 + 版本锁定下载）。节奏：注册 < 5 秒、5 秒
心跳、60 秒重新扫描；服务端在守护进程静默 45 秒后将其标记为离线，并在 7 天后删除它。
一个守护进程可服务多个空间。

## 应用升级

```bash
octo-daemon upgrade    # npm install -g @latest, then signals the process to restart
```

<Warning>
  `upgrade` 会重新安装并向进程发信号——一个 supervisor **必须**重新拉起它。pm2 默认这样做；
  在 systemd 下请设置 `Restart=always`。
</Warning>

## 升级平台本身

对于 server/客户端，请将 `mininglamposs/octo-*` 镜像固定到某个 release 标签（切勿用 `:latest`），
拉取新标签，然后滚动更新：`docker compose up -d`（Compose）或 `helm upgrade`（Kubernetes）。
在进行大版本升级前，请查阅 [备份与恢复](/zh/guides/operators/backup-and-restore)。
