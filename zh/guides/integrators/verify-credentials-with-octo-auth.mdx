如果你构建了一个位于 Octo 凭据之后的服务——一个适配器、一个配套 API、一个
Hocuspocus 文档服务器——请使用 **[`octo-auth`](https://github.com/Mininglamp-OSS/octo-auth)** 来
校验调用方，而不是自己造轮子。它以 **Go** 和 **TypeScript** 两种形式发行，行为完全一致，
并且这正是 `octo-server` 所强制执行的同一套契约。

## 安装

```bash
# Go（需要 Go 1.25+）
go get github.com/Mininglamp-OSS/octo-auth/go
```

<Note>
  TypeScript 包 `@mininglamp-oss/octo-auth`(纯 ESM,Node ≥ 20)**尚未发布**到公共 npm registry ——
  在发布之前请从仓库安装：
  ```bash
  npm install "git+https://github.com/Mininglamp-OSS/octo-auth.git#main:ts"
  ```
</Note>

## 校验一个请求

`MultiVerifier` 按凭据前缀分派（`uk_` → api-key、`bf_`/`app_` → bot、否则 →
session），因此你可以在一个接口之后处理全部三种领域：

```go
v := octoauth.NewFullMultiVerifier(octoauth.Options{
    BaseURL: "https://your-octo-instance.com",
})

// In your handler, or use the middleware below:
principal, err := v.Verify(ctx, token)
if errors.Is(err, octoauth.ErrInvalidCredential) {
    // 401 — do not leak the specific reason
}
```

只有 `BaseURL` 是必需的。一个可插拔的 **缓存**（默认是内存 LRU）、一个
**指标收集器**，以及一个 **日志器**，全部可注入。

## 使用中间件

`octo-auth` 附带中间件，因此你很少需要直接接触校验器：

| 语言 | 框架 |
|---|---|
| Go | `net/http`、`gin`、`echo`、`wkhttp` |
| TypeScript | `express`、`@hocuspocus/server`（`onAuthenticate` hook） |

```ts
import { newFullMultiVerifier, expressMiddleware } from "@mininglamp-oss/octo-auth";

const verifier = newFullMultiVerifier({ baseURL: process.env.OCTO_URL });
app.use(expressMiddleware({ verifier }));
```

<Warning>
  **失败即拒绝（Fail closed）。** 默认的错误映射器把基础设施故障视为拒绝，且每一种“坏
  令牌”原因都收敛为单一的 `AUTH_INVALID_CREDENTIAL`（401）——绝不要基于某个
  子原因来分支。对于智能体流量，SDK 会忽略客户端发送的任何 `X-Space-Id`（智能体的空间以
  服务端为权威）。参见[反枚举](/zh/concepts/security-and-auth)。
</Warning>

## 一次调用获取主体（principal）的上下文

在 session / api-key 校验时传入 `?include=context`（通过 `RequestIncludeContext`），即可获取
调用方的空间和其拥有的智能体，无需第二次往返。`verify-bot` 不接受它——
智能体的空间在服务端固定。
