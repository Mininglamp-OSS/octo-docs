**incoming webhook（入站 webhook）**让外部系统通过向一个带 token 的 URL 发起 POST，把消息推送进
某个 Octo **群组**或**话题**。该消息会以一个合成的 webhook 发送者（一个 `iwh_…` 身份）出现在频道中
——你无需运行智能体进程，也无需持有登录态。它是把 CI 结果、告警或第三方事件送入会话的最轻量方式。

<Info>
  incoming webhook 由 `octo-server`(`incomingwebhook` 模块)作为 **REST 功能**提供——它**不属于**
  [`octo-cli`](/zh/guides/bot-developers/drive-octo-with-cli)。`iwh_` 发送者是[数据模型](/zh/reference/data-model#横切枚举)
  中的凭证类型之一。
</Info>

## 认证机制

每个 webhook 有两部分：

- **`webhook_id`**——一个公开的 `iwh_…` 标识，出现在 URL 中（也是消息发送者的 id）。
- **`token`**——一个 256 位的密钥。服务端只存储它的 SHA-256 哈希并以常量时间比较,因此 token 只在创建时
  **出示一次**,之后不再展示。

URL 中的 token 是投递的**唯一凭证**——任何拿到完整 URL 的人都能推送消息,所以请像对待密码一样对待它。
所有认证失败都统一返回 `401`(服务端绝不透露是 id 还是 token 出错)。

## 创建一个 webhook

创建 webhook 是一个管理调用,要求调用者是目标群组的正常成员——机器人 token(`bf_…`)或用户凭证都可以。
它会返回 token 与投递 URL。

<Steps>
  <Step title="调用创建接口">
    ```bash
    export OCTO_API_BASE_URL="https://your-octo-instance.com"
    export OCTO_BOT_TOKEN="bf_your_token"

    curl -X POST "$OCTO_API_BASE_URL/v1/groups/$GROUP_NO/incoming-webhooks" \
      -H "Authorization: Bearer $OCTO_BOT_TOKEN" \
      -H "Content-Type: application/json" \
      -d '{"name":"CI alerts","allow_mention_bots":true}'
    ```

    字段:`name`(≤ 64 字符;省略则自动生成),以及提及策略——`allow_mention_all`、
    `allow_mention_bots`、`mention_uids`(见[提及](#提及))。`avatar` 仅群组管理员可设置。若要将目标
    定为某个**话题**而非整个群组,改用 `/v1/groups/$GROUP_NO/threads/$SHORT_ID/incoming-webhooks`。
  </Step>

  <Step title="保存 token 与 URL">
    响应会包含仅出示一次的 `token`、一个原生 `url`,以及一个 `urls` 映射,其中每个适配器一项
    (`native`、`github`、`gitlab`、`feishu`、`wecom`)。

    <Warning>
      返回的 URL 是**不含域名的路径**——请自行拼接你的 Octo 基础域名。而且 `token` 之后不会再展示:
      请现在就保存,或稍后[轮换](#管理-webhook)。
    </Warning>
  </Step>
</Steps>

## 发送你的第一个事件

向裸的 `webhook_id/token` 路径 POST **原生**载荷:

```bash
curl -X POST "$OCTO_API_BASE_URL/v1/incoming-webhooks/$WEBHOOK_ID/$TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"msg_type":"text","content":"Deploy #4123 succeeded ✅"}'
```

- **`msg_type`**——`text`(默认)、`richtext`(一个 `blocks` 数组)或 `card`(位于 `card` 的
  Adaptive Card)。其他取值一律以 `400` 拒绝。
- **`content`**——文本正文;`text` 可作为 Slack 风格的别名。
- **`username` / `avatar_url`**——覆盖发送者展示信息,但**仅当该 webhook 的创建者当前是群组管理员**时
  生效(否则使用已配置的名称/头像)。

<Note>
  限制:原生请求体上限为 8 KiB,`content` 上限为 4000 字符;第三方适配器(GitHub/GitLab)可接受
  至多 1 MiB。投递按 webhook 限流——突发流量会得到带 `Retry-After` 头的 `429`。
</Note>

## 第三方适配器

对于会发出自有 webhook 格式的系统,在 URL 后追加提供方名称——适配器会把该提供方的载荷转换为 Octo
消息。把该 URL 粘贴进对应系统的 webhook 设置即可。

| 提供方 | URL 后缀 | 说明 |
|---|---|---|
| **Native（原生）** | *(无)* | 你自己的 `msg_type` / `content` 载荷(见上)。 |
| **GitHub** | `/github` | 粘贴为仓库的 **Payload URL**。 |
| **GitLab** | `/gitlab` | 同时把 webhook 的 **Secret Token** 设为同一个 URL token——它会通过 `X-Gitlab-Token` 头校验。 |
| **飞书（Feishu）** | `/feishu` | 粘贴为自定义机器人 URL。URL token 是唯一凭证——飞书自带的 `timestamp`/`sign` 会被忽略。 |
| **企业微信（WeCom）** | `/wecom` | 使用企业微信群机器人的消息格式。 |

例如,一个 GitHub 仓库 webhook 应指向:

```
https://your-octo-instance.com/v1/incoming-webhooks/iwh_…/…/github
```

## 提及

`@` 提及由 webhook 的**配置**驱动,而绝非投递请求体(载荷中的 `mention` 字段会被忽略)。请在
创建/更新时设置:

- **`mention_uids`**——要 @ 的具体用户/智能体(至多 50 个)。
- **`allow_mention_bots`**——允许 `@所有AI`(所有智能体)。
- **`allow_mention_all`**——允许 `@所有人`;广播式提及还会受群组策略进一步限制。

对于 `text` 消息,服务端会自动前置提及气泡并生成对应的线级 mention 实体。

## 管理 webhook

所有管理路由都位于群组(或话题)作用域下,并且还提供一个供机器人 token 调用的 `/v1/bot/groups/…` 入口:

| 方法与路径 | 用途 |
|---|---|
| `POST /v1/groups/{group_no}/incoming-webhooks` | 创建(返回 token + URL,仅一次)。 |
| `GET /v1/groups/{group_no}/incoming-webhooks` | 列出作用域内的 webhook(绝不回显 token)。 |
| `PUT …/incoming-webhooks/{webhook_id}` | 更新名称、头像、状态或提及配置。 |
| `DELETE …/incoming-webhooks/{webhook_id}` | 禁用(软删除)。 |
| `POST …/incoming-webhooks/{webhook_id}/regenerate` | 轮换 token(返回一个新的)。 |
| `GET …/incoming-webhooks/{webhook_id}/deliveries?limit=50` | 最近的投递记录(成功 / 失败 / 跳过)。 |
| `POST …/incoming-webhooks/{webhook_id}/test` | 发送一条端到端测试消息。 |

<Tip>
  webhook 没反应?先调用 `/test` 确认整条链路,再读 `/deliveries`——每条记录都带有状态和一个
  `reason`(例如被拒的 `msg_type`、认证失败,或限流跳过)。
</Tip>
