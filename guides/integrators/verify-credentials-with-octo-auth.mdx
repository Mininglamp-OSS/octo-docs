If you build a service that sits behind Octo credentials — an adapter, a companion API, a
Hocuspocus doc server — use **[`octo-auth`](https://github.com/Mininglamp-OSS/octo-auth)** to
verify callers instead of rolling your own. It ships in **Go** and **TypeScript** with identical
behavior, and it's the same contract `octo-server` enforces.

## Install

```bash
# Go (requires Go 1.25+)
go get github.com/Mininglamp-OSS/octo-auth/go
```

<Note>
  The TypeScript package `@mininglamp-oss/octo-auth` (pure ESM, Node ≥ 20) is **not yet on the
  public npm registry** — install it from the repository until it's published:
  ```bash
  npm install "git+https://github.com/Mininglamp-OSS/octo-auth.git#main:ts"
  ```
</Note>

## Verify a request

The `MultiVerifier` dispatches by credential prefix (`uk_` → api-key, `bf_`/`app_` → bot, else →
session), so you handle all three realms behind one interface:

```go
v := octoauth.NewFullMultiVerifier(octoauth.Options{
    BaseURL: "https://your-octo-instance.com",
})

// In your handler, or use the middleware below:
principal, err := v.Verify(ctx, token)
if errors.Is(err, octoauth.ErrInvalidCredential) {
    // 401 — do not leak the specific reason
}
```

Only `BaseURL` is required. A pluggable **cache** (in-memory LRU by default), a
**metrics collector**, and a **logger** are all injectable.

## Use the middleware

`octo-auth` ships middleware so you rarely touch the verifier directly:

| Language | Frameworks |
|---|---|
| Go | `net/http`, `gin`, `echo`, `wkhttp` |
| TypeScript | `express`, `@hocuspocus/server` (`onAuthenticate` hook) |

```ts
import { newFullMultiVerifier, expressMiddleware } from "@mininglamp-oss/octo-auth";

const verifier = newFullMultiVerifier({ baseURL: process.env.OCTO_URL });
app.use(expressMiddleware({ verifier }));
```

<Warning>
  **Fail closed.** The default error mapper treats infra failures as denials, and every "bad
  token" reason collapses to a single `AUTH_INVALID_CREDENTIAL` (401) — never branch on a
  sub-reason. For agent traffic, the SDK ignores any client-sent `X-Space-Id` (the agent's space is
  server-authoritative). See [anti-enumeration](/concepts/security-and-auth).
</Warning>

## Get the principal's context in one call

Pass `?include=context` (via `RequestIncludeContext`) on session / api-key verification to get
the caller's spaces and owned agents without a second round trip. `verify-bot` doesn't accept it —
an agent's space is fixed server-side.
